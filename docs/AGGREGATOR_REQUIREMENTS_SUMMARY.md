# Aggregator 产品需求整理

基于 [Aggregator产品需求结构 (1).pdf](/Users/weili/Downloads/Aggregator产品需求结构%20%281%29.pdf) 提炼整理。

## 1. 产品目标

Aggregator 是一个面向开发者和企业的 AI 模型聚合平台，对标 InfronAI、ZenMux、OpenRouter。

核心目标：

- 让用户快速理解“一个 API 接入多个 LLM”
- 让开发者低成本接入并获取 API Key
- 让平台具备多 Provider 接入、统一网关、智能路由能力
- 让调用、计量、计费、充值和账单形成完整闭环
- 为后续团队协作、权限分层、调试排障预留扩展空间

## 2. MVP 范围

MVP 的核心不是一次做完所有后台，而是优先跑通以下主链路：

`官网获客 -> 注册登录 -> 获取 API Key -> 调用模型 -> 统计消耗 -> 余额/充值 -> 账单明细`

只要这条链路稳定可用，就已经具备上线验证价值。

  - 条件筛选
  - 价格表
  - 模型能力标签
  - 模型详情
- `Pricing`
  - 价格说明
  - 模型价格表

  - 使用成本示例

- `Documentation`
  - Overview
  - Quickstart
  - API Reference

  - Pricing

- CDN / 基础性能优化

- 会话管理

目标：

### 3.4 个人中心

- 个人信息管理

  - 查看

  - 撤销

### 3.5 权限与 API Key 最小闭环

  - Admin
  - Member
  - Guest
- API Key 管理
  - 创建唯一 Key

- 原始需求把“角色定义”列为 P0

### 3.6 Provider 接入系统

- Provider 注册
- 连接配置
- 健康检查

- 能接入新的 LLM Provider
- 统一调用方式
- 具备最基本的可用性检测和回滚能力

### 3.7 模型管理

- 模型列表
- 模型能力标签
- 模型详情
- 模型状态

说明：

- 这部分既服务前台展示，也服务后台路由和可用性判断

### 3.8 Router 系统

- 请求路由
- Fallback 机制
- 负载均衡
- 路由策略管理

目标：

- 在多 Provider / 多实例场景下稳定分发请求
- 避免单点故障
- 支撑后续按健康度、延迟、成本进行策略调度

### 3.9 API Gateway

- 统一接口
- 协议转换
- 权限校验
- 请求日志
- 接口限流

说明：

- 协议优先支持 OpenAI 和 Anthropic
- 这是平台最关键的底层能力

### 3.10 实时计量与监控

- Token 消耗
- 请求数
- TTFT
- Latency
- 成功率 / 错误率
- 消耗趋势
- 异常告警

目标：

- 让平台能够实时看到每次调用的成本和质量
- 支撑后续健康监控、路由优化、账单核算

### 3.11 钱包、充值和账单

- 支付接入
- 余额管理
- 充值操作
- 充值历史
- 费用明细
- 报表导出

说明：

- 原文里支付接入直接标成 P0，但批注倾向先从 ToB 企业级场景切入
- 成本趋势图可放后续增强

### 3.12 请求状态可见

- 调用状态展示

说明：

- 在开发者调试相关能力里，“调用状态”被标成 P0
- 这意味着哪怕先不做完整请求日志中心，也要先让用户知道请求是否成功、失败以及基本状态

## 5. 建议实施顺序

### 里程碑 1：获客与首次调用

- Landing Page
- Models
- Pricing
- Documentation
- 注册登录
- API Key 创建
- 统一 API 调用链路

交付结果：

- 用户可完成注册
- 用户可拿到 API Key
- 用户可完成第一次模型调用

### 里程碑 2：网关能力成型

- Provider 接入
- 模型管理
- Router
- Gateway
- 权限校验
- 限流

交付结果：

- 平台具备真正的聚合器能力
- 能稳定对接多个 Provider / 实例

### 里程碑 3：计量与商业闭环

- Token 计量
- 请求监控
- 成功率 / 延迟指标
- 余额
- 充值
- 费用明细
- 报表导出

交付结果：

- 平台具备基础商业化能力
- 用户可以理解费用并完成结算

### 里程碑 4：协作与高级体验

- 团队管理
- 更细权限体系
- 审计日志
- Playground
- 完整请求日志追踪

交付结果：

- 面向团队和企业的产品能力补齐
- 平台更适合长期运维和排障

## 6. 待澄清项

以下问题在原始 PDF 批注中多次出现，建议在正式排期前明确：

- 国内认证到底优先手机号还是可先用邮箱登录过渡
- 忘记密码是否纳入 MVP，取决于基础框架支持成本
- Landing Page 是否需要加入 FAQ 和 skill 安装示例
- Documentation 是否需要首批补全错误码列表
- MVP 支持哪些模型类型
  - 仅文本
  - 还是包含生图、TTS、Embedding
- “最优模型”定义是什么
  - 同模型下最健康实例
  - 最低延迟
  - 最低成本
  - 综合评分
- Fallback 全失败时返回策略是什么
  - 原始批注中提到过 `429`，但这需要和标准错误语义再确认
- Provider 监控指标是否包含 TTFT、TPOT、负载压力等更细粒度能力
- 权限体系是否需要和 credit / 垫付规则绑定
- 请求日志保留策略是什么
  - 全量存储
  - 仅当月
  - 按用户 / 组织分层
- 支付是否先只做 ToB 企业客户

## 7. 一句话结论

这份需求的真正 P0 是：先交付一个“能卖、能接、能调度、能计量、能收费”的 AI 聚合平台 MVP；团队协作、深度权限和高级调试能力放到 P1。
